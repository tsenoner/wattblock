# Wattblock v0.1.1 — Empty Names, Navigation, and Finished-Game Reload

**Date:** 2026-05-11
**Status:** Approved
**Branch:** `feature/v0.1.1-fixes`
**Builds on:** `docs/superpowers/specs/2026-05-11-wattblock-design.md`

## Goal

Fix three v0.1.0 usability bugs surfaced after first use:

1. The **Starten** button is disabled until both team names are entered, but the user wants to be able to play without naming the teams.
2. There is no visible affordance to return to the setup screen from the scoring screen. (The SPIELZIEL block at the top is tappable but doesn't look like a button.)
3. After a game finishes and the browser is closed, the next launch resumes on the winner banner. The user wants finished games to be "consumed" so the next launch is a fresh setup.

## Non-Goals

- No state-schema migration. The persisted shape stays identical.
- No new persistence options (no toggle for "remember last game").
- No "are you sure?" confirmation when the user navigates back to setup via the gear icon mid-match.
- No animation / transition polish.

## Behaviour Changes

### 1. Empty team names default to "Team 1" / "Team 2" (display only)

- `renderColumn(state, team)` shows the column header as `state.setup.teamX || "Team N"` (N = 1 for team A, 2 for team B). The empty string is still what's stored.
- The winner banner (`renderScoring`) uses the same fallback for the winner's name.
- Setup screen text inputs get `placeholder="Team 1"` / `placeholder="Team 2"` so the implicit default is visible to the user before they type anything.
- The **Starten** button no longer carries `disabled` when names are empty. Start is always allowed; the fallback names appear if the user starts with no names typed.
- If the user types a name, then later clears the input (mid-match via gear → setup), the column header reverts to "Team 1" / "Team 2". Symmetric and explained by the rule "display empty as default".

### 2. Gear icon for back-to-setup

- A new button is rendered in the top-right of the scoring screen with an inline SVG gear icon. Markup roughly:

  ```html
  <button type="button" class="scoring__settings" data-action="open-setup"
          aria-label="Einstellungen">
    <svg viewBox="0 0 24 24" …>
      <!-- minimal gear glyph -->
    </svg>
  </button>
  ```

- The SPIELZIEL block reverts to a plain non-interactive `<div>` (no `data-action`, no `cursor: pointer`). It becomes pure information.
- The gear button uses `position: absolute; top: 8px; right: 8px` over the scoring layout, sized ~36 × 36 px, neutral fill, and respects light/dark.
- Tap → `data-action="open-setup"` → existing `openSetup` action → setup view, all current state preserved.

### 3. Auto-consume a finished game on load

- A new pure function in `state.ts`:

  ```ts
  export function resumeFromPersisted(state: AppState): AppState {
    return winnerOf(state) !== null ? newMatch(state) : state;
  }
  ```

  This is purely defined in terms of existing actions (`winnerOf` + `newMatch`), so it carries no new behaviour rules.

- `main.ts` boot sequence becomes:

  ```ts
  const loaded = load();
  let state = resumeFromPersisted(loaded);
  if (state !== loaded) save(state); // only persist if we actually advanced past a finished game
  render(state, root);
  ```

  The reference-equality check avoids an unnecessary write on every cold start.

- Effect: if `localStorage` contained a state with a winner (i.e. the last session ended on a winner banner), the app loads as if the user had tapped the banner and then the gear icon — that is, scores cleared, names + target preserved, setup view shown.

## State Model

Unchanged. `AppState`, `ScoreEntry`, `Setup`, `INITIAL_STATE` all stay as they were. No new fields, no schema version bump.

## File Changes

| Path | Change |
| --- | --- |
| `src/state.ts` | Add `resumeFromPersisted` function. |
| `src/state.test.ts` | Add 2 tests for `resumeFromPersisted` (identity when no winner; equals `newMatch(state)` when winner). |
| `src/render.ts` | Column header fallback to "Team 1" / "Team 2"; winner banner uses same fallback; setup inputs get `placeholder`; drop `disabled` from Starten; SPIELZIEL block → non-interactive div; new gear button. |
| `src/app.css` | Style `.scoring__settings` (top-right gear). Reset cursor/padding on the (now plain) target block. |
| `src/main.ts` | Use `resumeFromPersisted(load())` at boot. Persist the resumed state so the next load is idempotent. |

No new files, no removed files.

## Testing

- **Unit tests** — add `resumeFromPersisted` tests to `src/state.test.ts`. Existing 23 tests untouched.
- **Manual smoke test** (added to README checklist):
  - Start a game without typing any names — both columns show "Team 1" / "Team 2"; Starten is enabled.
  - Type "Tobias" / "Anna", start, return to setup via the gear icon — names are preserved in the inputs.
  - Win a match, close the tab, reopen — app lands on setup with names + target preserved, scores empty.
  - In-session banner behaviour is unchanged: the winner banner shows during the same session and a tap on it clears scores and jumps to scoring.

## Out of Scope (deferred)

- A "Discard match" confirmation when leaving the scoring screen mid-game via the gear icon. Current behaviour: state is preserved across that navigation. Adding a confirmation step is a separate UX decision.
- Storage schema version field. Premature for v0.1.1; the shape hasn't changed.

## Open Questions

None. All three behaviours are fully specified above.
