# Wattblock v0.1.2 ÔÇö Mobile PWA Fixes and UX Polish

**Date:** 2026-05-11
**Status:** Approved
**Branch:** `feature/v0.1.2-mobile-and-ux`
**Builds on:** `docs/superpowers/specs/2026-05-11-v0.1.1-empty-names-and-navigation.md`

## Goal

Fix mobile PWA viewport bugs that surfaced after installing the app on a phone, and ship four user-requested UX improvements:

1. **Bug:** Pinch-zoom, double-tap zoom, and rubber-band scroll all happen when the PWA should behave as a locked single-screen app.
2. **Theme toggle:** Let the user override the system colour scheme via a single icon button that cycles `auto ÔåÆ light ÔåÆ dark ÔåÆ auto`.
3. **Single global undo + redo** instead of one undo button per team rail.
4. **Mid-game settings labels:** Replace the single `Starten` button with `Spiel fortsetzen` + `Neues Spiel` when a game is in progress.
5. **Revanche banner:** Replace the existing `"<Team> gewinnen Ôå╗ neues Spiel"` text with a two-line `"<Team> gewinnt"` / `"Revanche Ôå╗"` banner.

## Non-Goals

- No `user-scalable=no` in the viewport meta. The fix blocks the *double-tap* zoom gesture (via `touch-action: manipulation`), but page-level pinch-zoom remains available as an accessibility affordance.
- No "are you sure?" confirmation on **Neues Spiel** mid-game. Current behaviour (silent discard) is unchanged.
- No persistence schema version bump. The new `undone` field is treated as optional by the storage guard.
- No history navigation beyond a single linear redo stack.
- No animated theme transitions.

## 1. Mobile viewport fixes

### Root causes identified

| Symptom | Cause |
| --- | --- |
| Double-tap zoom on score buttons | No `touch-action: manipulation` on any tappable element. |
| Rubber-band / overscroll past content | `html, body` have no `overscroll-behavior`. |
| Page scrolls vertically when content fits one screen | `body { min-height: 100dvh }` allows scroll; nothing locks the document height. |
| Layout jumps with iOS chrome (URL bar shows/hides) | `100dvh` is a *dynamic* unit. For an in-PWA standalone layout, the **smallest viewport** (`100svh`) is more stable. |
| Gear icon risks overlapping the iPhone notch | `.scoring__settings { top: 4px; right: 4px }` is too tight; needs safe-area cushion. |
| Long-press shows iOS text-selection menu over buttons | No `user-select: none` on UI chrome. |

### Changes

1. **`index.html` ÔÇö viewport meta**

   Replace:
   ```html
   <meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover" />
   ```
   with:
   ```html
   <meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover, interactive-widget=resizes-content" />
   ```

   `interactive-widget=resizes-content` tells iOS/Android to push the layout up when the on-screen keyboard appears, rather than overlaying it. No `user-scalable` change.

2. **`src/app.css` ÔÇö document lock**

   ```css
   html, body {
     height: 100svh;
     overflow: hidden;
     overscroll-behavior: none;
   }
   ```

   `height` (not `min-height`) plus `overflow: hidden` means the document is exactly one viewport tall and cannot scroll. `overscroll-behavior: none` defends against rubber-band on touch.

3. **`src/app.css` ÔÇö `#app` becomes the inner frame**

   ```css
   #app {
     max-width: 540px;
     margin: 0 auto;
     padding: 16px;
     height: 100%;
     overflow: hidden;
     display: flex;
     flex-direction: column;
   }
   ```

   Flex column so the inner `<main>` can stretch to fill, with overflow contained.

4. **`src/app.css` ÔÇö chrome tap behaviour**

   Add to every interactive element selector (`.chip`, `.setup__input`, `.setup__start`, `.setup__newmatch`, `.rail__btn`, `.scoring__settings`, `.banner`, `.theme-toggle`, `.history__btn`):
   ```css
   touch-action: manipulation;
   -webkit-user-select: none;
   user-select: none;
   ```
   Keep `user-select: text` on `.setup__input` (text fields should remain selectable).

5. **`src/app.css` ÔÇö scoring layout uses svh**

   Replace `.scoring { min-height: calc(100dvh - 32px) }` with:
   ```css
   .scoring {
     display: grid;
     grid-template-rows: auto 1fr auto auto;  /* header, body, history bar, banner */
     gap: 8px;
     flex: 1;
     min-height: 0;                            /* allow children to shrink */
     position: relative;
   }
   ```
   The `flex: 1` lets `.scoring` fill the parent `#app` flex column; `grid-template-rows` now reserves space for the new history bar (Section 3) and the banner (Section 5).

6. **`src/app.css` ÔÇö gear safe-area**

   ```css
   .scoring__settings {
     top: max(4px, env(safe-area-inset-top));
     right: max(8px, env(safe-area-inset-right));
     /* ÔÇªexisting stylesÔÇª */
   }
   ```

   Note: the body already applies `env(safe-area-inset-top)` padding, so this `max()` is a defence-in-depth rather than a hard requirement. It also keeps the icon away from the notch when the body padding is small.

7. **`src/app.css` ÔÇö long lists scroll within the column**

   With the document locked, a very long score list (many `-2` retries against `target=18`) could clip. Add:
   ```css
   .col__scores { overflow-y: auto; }
   ```
   This makes the column's own score list scroll, not the page.

### Verification

- Playwright with iPhone-14-class viewport (`webkit` engine in `device.iPhone 14`) drives the golden path: setup ÔåÆ score to 11 ÔåÆ win. Assertions: `document.documentElement.scrollHeight === window.innerHeight`, no rubber-band evidence.
- Manual verification on the user's installed PWA (iOS Add-to-Home-Screen).

## 2. Theme toggle (auto ÔåÆ light ÔåÆ dark)

### State

A new module `src/theme.ts`:

```ts
export type Theme = "auto" | "light" | "dark";
export const THEME_KEY = "wattblock:theme";

export function loadTheme(): Theme {
  const raw = localStorage.getItem(THEME_KEY);
  return raw === "light" || raw === "dark" || raw === "auto" ? raw : "auto";
}

export function saveTheme(t: Theme): void {
  localStorage.setItem(THEME_KEY, t);
}

export function nextTheme(t: Theme): Theme {
  return t === "auto" ? "light" : t === "light" ? "dark" : "auto";
}

export function applyTheme(t: Theme): void {
  const root = document.documentElement;
  if (t === "auto") root.removeAttribute("data-theme");
  else root.setAttribute("data-theme", t);
}
```

Theme lives **separately** from `AppState`. It is a user preference, not match state, and changing it does not invalidate persisted matches.

### CSS

```css
:root { /* light tokens ÔÇö existing */ }

@media (prefers-color-scheme: dark) {
  :root { /* dark tokens ÔÇö existing */ }
}

:root[data-theme="light"] { /* light tokens (override system) */ }
:root[data-theme="dark"]  { /* dark tokens (override system) */ }
```

The light/dark token blocks must be authored once and referenced from both the media query and the attribute selector. Use CSS custom properties as the single source: keep current `:root` for light, the existing `prefers-color-scheme: dark` block for system-dark, and add two new attribute-selector blocks that re-state the same tokens. This is duplication but it's localised and trivial to audit.

### Toggle UI

A single icon button rendered in **both** the setup view and the scoring view:

```html
<button type="button" class="theme-toggle" data-action="cycle-theme"
        aria-label="Theme: auto | light | dark"
        title="Theme">
  <svg viewBox="0 0 24 24" ...><!-- glyph reflects current Theme --></svg>
</button>
```

**Glyphs** (inline SVG, all 24├ù24):
- `auto` ÔÇö a circle with the left half filled and the right half outlined (the "contrast" glyph).
- `light` ÔÇö sun (filled circle + 8 short straight rays).
- `dark` ÔÇö crescent moon (one filled path).

**Placement:**
- **Setup screen:** top-right of `<main class="setup">`. Setup currently has no top-bar; add a flex header row containing only the toggle, right-aligned.
- **Scoring screen:** to the **left** of the gear, so the top-right of the scoring screen shows two icons side by side. Both share an `.scoring__chrome` row with `position: absolute; top: max(4px, env(safe-area-inset-top)); right: max(8px, env(safe-area-inset-right))`.

### Boot order

`main.ts` runs `applyTheme(loadTheme())` **before** the first `render()`, so the page paints in the correct theme.

### Action

```ts
case "cycle-theme": {
  const t = nextTheme(loadTheme());
  saveTheme(t);
  applyTheme(t);
  render(state, root!);  // re-render so the glyph updates
  break;
}
```

Theme state is fetched fresh via `loadTheme()` on click rather than threaded through `AppState`, keeping `render()` pure of theme concerns.

### `render()` reads theme

`render.ts` imports `loadTheme` from `./theme` and calls it when emitting the toggle button (in both `renderSetup` and `renderScoring`). This couples `render.ts` to localStorage, but the boundary is a single function call per render, and avoids threading theme through the rest of the state pipeline.

Alternative considered and rejected: extend `AppState` with `theme`, or pass a view-model wrapper to `render`. YAGNI for one icon.

## 3. Single global undo + redo

### State model

`AppState` gains one field:

```ts
export interface AppState {
  setup: Setup;
  scores: ScoreEntry[];
  undone: ScoreEntry[];   // NEW ÔÇö redo stack, LIFO
  view: "setup" | "scoring";
}

export const INITIAL_STATE: AppState = {
  setup: { target: 15, teamA: "", teamB: "" },
  scores: [],
  undone: [],
  view: "setup",
};
```

### Reducer changes

```ts
export function addPoint(state: AppState, team: TeamId, value: number): AppState {
  if (winnerOf(state) !== null) return state;
  return { ...state, scores: [...state.scores, { team, value }], undone: [] };
}

export function undo(state: AppState): AppState {
  if (state.scores.length === 0) return state;
  const last = state.scores[state.scores.length - 1];
  return {
    ...state,
    scores: state.scores.slice(0, -1),
    undone: [...state.undone, last],
  };
}

export function redo(state: AppState): AppState {
  if (state.undone.length === 0) return state;
  const last = state.undone[state.undone.length - 1];
  return {
    ...state,
    scores: [...state.scores, last],
    undone: state.undone.slice(0, -1),
  };
}
```

### Semantic notes

- `undo()` no longer takes a `team` argument. It pops the **most recent** entry regardless of team.
- `addPoint()` always clears the redo stack ÔÇö once you take a new action, the old "redoable" future is lost. This is standard editor behaviour.
- `undo()` works even when there is a winner (you can rewind out of a win ÔÇö useful if a point was mis-entered as the winning one). After undo, `winnerOf` returns `null` again and `addPoint` is unblocked.
- `redo()` *is* blocked when there is a winner ÔÇö pushing the winning entry back creates a winning state, which is fine. So redo is *not* blocked by `winnerOf`. (No special case needed; the function just pushes.)
- `newMatch()` resets both `scores` and `undone` to `[]`.
- `resumeFromPersisted()` is unaffected ÔÇö when there's a winner it calls `newMatch`, which now also clears `undone`.

### UI

**Rails:** remove the undo button. Each rail is now five buttons: `2 / 3 / 4 / 5 / ÔêÆ2`.

### Storage migration

`storage.ts` guard tolerates `undone` missing (old persisted state from v0.1.1 or earlier):

```ts
function isAppState(value: unknown): value is AppState {
  // ÔÇªexisting checksÔÇª
  // If undone present, it must be an array. If absent, that's fine ÔÇö we'll fill it in.
  if (v.undone !== undefined && !Array.isArray(v.undone)) return false;
  return true;
}

export function load(): AppState {
  try {
    const raw = localStorage.getItem(STORAGE_KEY);
    if (raw === null) return INITIAL_STATE;
    const parsed: unknown = JSON.parse(raw);
    if (!isAppState(parsed)) return INITIAL_STATE;
    return { ...parsed, undone: parsed.undone ?? [] };
  } catch {
    return INITIAL_STATE;
  }
}
```

No `STORAGE_KEY` bump.

## 4. Mid-game settings buttons

### Behaviour

The setup view renders buttons depending on whether a game is in progress:

- **`scores.length === 0`** (fresh state, no game): single primary button **`Starten`** ÔåÆ `start(state)`.
- **`scores.length > 0`** (game in progress, with or without a winner): replace `Starten` with two stacked buttons:
  - Primary: **`Spiel fortsetzen`** ÔåÆ `start(state)` (back to scoring view, scores intact).
  - Secondary: **`Neues Spiel`** ÔåÆ `newMatch(state)` (clears scores + undone, names + target preserved, stays in setup view).

The user can edit names and target before tapping either button. Existing input handlers (`team-a`, `team-b`, `custom-target`, `preset-target`) are unchanged.

### Render

```ts
function renderSetup(state: AppState): string {
  const hasGame = state.scores.length > 0;
  const startBlock = hasGame
    ? `
      <button type="button" class="setup__start" data-action="start">Spiel fortsetzen</button>
      <button type="button" class="setup__newmatch" data-action="new-match">Neues Spiel</button>
    `
    : `<button type="button" class="setup__start" data-action="start">Starten</button>`;
  return `
    <main class="setup">
      <!-- theme toggle row -->
      <!-- existing inputs/chips -->
      ${startBlock}
    </main>
  `;
}
```

The old `${state.scores.length > 0 ? `<button ÔÇª data-action="new-match">ÔÇª` : ""}` conditional is replaced by the new branch above (no longer mixed in below `Starten`).

### Actions

No new `data-action` values needed: `start` and `new-match` already exist in `main.ts`.
